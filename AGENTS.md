# AGENTS.md

## Назначение проекта

`tunnelctl` — кроссплатформенная CLI-утилита на Go для устойчивого локального SOCKS5-прокси поверх системного SSH. Поддерживаемые среды: Linux, Windows и Android/Termux.

Источники истины по ожидаемому поведению: актуальная ветка `main`, этот файл, `README.md`, относящиеся к задаче исходники, тесты и документация. Реализацию подтверждают код и проверки, а не только описание.

## Архитектурные инварианты

- Системный `ssh` запускается как управляемый дочерний процесс `tunnelctl`.
- Не использовать скрытый режим `ssh -f`.
- SOCKS по умолчанию слушает только loopback; внешний bind допускается только при явной настройке.
- Работоспособность SOCKS подтверждается реальным HTTP/HTTPS-запросом через прокси. Открытый порт или живой процесс сами по себе недостаточны.
- `connect --watch` владеет дочерним `ssh`, выполняет health-check, backoff, переподключение и failover.
- `status`, `switch`, `stop` и `restart` взаимодействуют с работающим экземпляром через локальный управляющий канал текущего пользователя.
- Нельзя завершать посторонний процесс только потому, что он занял ожидаемый SOCKS-порт.
- Платформенный автозапуск отвечает только за запуск и состояние внешнего процесса. Health-check, backoff, failover и переключение остаются внутри `tunnelctl`.
- Не повышать привилегии скрыто и не вызывать `sudo` автоматически.
- Изменения должны сохранять совместимость Linux, Windows и Termux либо явно документировать платформенное ограничение.

## Карта репозитория

- `cmd/tunnelctl/` — точка входа CLI, дополнительные команды и форматирование верхнего уровня.
- `internal/app/` — основные команды CLI; версия приложения хранится в `internal/app/app.go`.
- `internal/config/` — схема, загрузка, миграции и валидация `tunnelctl.json`.
- `internal/supervisor/` — владение дочерним SSH, управляющий канал, state, switch, stop, restart и failover.
- `internal/sshproxy/` — формирование запуска SSH и проверка SOCKS реальным запросом.
- `internal/autostart/` — systemd, Windows Task Scheduler и Termux:Boot.
- `internal/bootstrap/` — первичная настройка и импорт команд `ssh -D`.
- `internal/logx/`, `internal/console/` — журналирование и операционный вывод.
- `internal/paths/` — платформенные пути конфигурации, state, логов и управляющего канала.
- `docs/` — дополнительная документация, включая `docs/VERSIONING.md`.
- `.github/workflows/` — обязательные автоматические проверки.

## Порядок работы

Перед изменениями:

1. Получить актуальное состояние `main` и последние коммиты.
2. Проверить рабочее дерево и не затрагивать посторонние изменения.
3. Прочитать `AGENTS.md`, `README.md`, относящиеся к задаче исходники, тесты и документацию.
4. Для незнакомой команды сначала изучить `--help`, dry-run, код и тесты.

При реализации:

- Вносить минимальные изменения и не форматировать соседний код без необходимости.
- Не менять архитектуру вне задачи и не добавлять зависимости без обоснования.
- Сохранять обратную совместимость, если задача явно не требует иного.
- Для изменяющего действия понимать цель, риск, обратимость и способ проверки.
- При неожиданном результате остановить дальнейшие изменяющие действия и оценить последствия.

## Безопасность

- Не добавлять пароли, приватные ключи, токены, адреса частных серверов и другие секреты в код, тесты, логи, документацию, issues или историю Git.
- Не выводить найденный секрет; сообщать только тип нарушения и безопасное местоположение.
- State, диагностика и ошибки не должны раскрывать SSH-пароли, ключи и лишние параметры подключения.
- В тестах использовать только искусственные значения.
- Проверять формирование и экранирование аргументов `ssh`, подмену путей и исполняемых файлов, права на конфигурацию, state, логи и управляющий канал.
- Учитывать различия Windows, Unix и Termux, особенно ACL/права, Job Objects, сокеты, named pipes и автозапуск.

## Версионирование

`tunnelctl` использует три независимые версии:

1. версия приложения — `app.Version` в `internal/app/app.go`;
2. версия управляющего протокола и state — в `internal/supervisor/`;
3. версия схемы конфигурации — `config.CurrentSchemaVersion` в `internal/config/`.

Правила изменения версии приложения:

- Использовать Semantic Versioning: `MAJOR.MINOR.PATCH`.
- `PATCH` повышать для совместимых исправлений без новой пользовательской функциональности.
- `MINOR` повышать для новых обратно совместимых команд, возможностей или заметных изменений поведения.
- `MAJOR` повышать при намеренно несовместимых изменениях CLI, конфигурации или поведения.
- Любой набор изменений, предназначенный для выпуска и меняющий пользовательское поведение, должен включать осознанное решение о версии до merge. Нельзя оставлять прежнюю версию по умолчанию.
- Версию приложения менять в одном месте — `internal/app/app.go`; не дублировать строковый номер в других пакетах.
- Версию управляющего протокола повышать только при несовместимом изменении формата запросов, ответов или state и сопровождать тестами совместимости/отказа.
- Версию схемы конфигурации повышать только при изменении формата `tunnelctl.json`; обязательно добавить миграцию или явный безопасный отказ для неподдерживаемой схемы.
- После изменения проверить `tunnelctl version`, `doctor`, запись версии в журнал/state и отображение версии работающего экземпляра в `status`.
- Документацию о совместимости и обновлении поддерживать синхронно с кодом.

## Проверки

Минимальный набор перед публикацией:

```bash
gofmt -w .
go test ./...
go vet ./...
go build ./cmd/tunnelctl
GOOS=linux GOARCH=amd64 go build ./cmd/tunnelctl
GOOS=windows GOARCH=amd64 go build ./cmd/tunnelctl
GOOS=android GOARCH=arm64 go build ./cmd/tunnelctl
```

Дополнительно запускать относящиеся к изменению интеграционные и платформенные проверки. Проверять не только код возврата, но и ожидаемое поведение:

- создание и завершение принадлежащего `tunnelctl` дочернего SSH;
- освобождение SOCKS-порта без завершения чужих процессов;
- health-check реальным запросом через SOCKS;
- переподключение, backoff и failover;
- управляющий канал, `status`, `switch`, `stop` и `restart`;
- конфигурацию, миграции, state и логи;
- автозапуск и dry-run;
- отсутствие секретов, conflict markers и побочных изменений.

Если платформенную проверку выполнить невозможно, указать это явно и не объявлять соответствующее поведение подтверждённым.

## Git и публикация

- Один логический набор изменений — один коммит, если задача не требует иного.
- Сообщения коммитов писать кратко и по-русски.
- Перед коммитом повторно проверить итоговый diff, состав файлов, форматирование, локальные пути, секреты и conflict markers.
- Не переписывать историю, не выполнять force-push и не удалять ветки или теги без прямого указания.
- Публиковать изменения в отдельной ветке через PR, если пользователь не указал иной порядок.
- Не утверждать, что коммит, push, PR, тест или исправление выполнены, без подтверждения инструментами.
