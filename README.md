# tunnelctl

`tunnelctl` — кроссплатформенная CLI-утилита для устойчивого локального SOCKS5-прокси поверх SSH.

Целевые системы:

- Ubuntu и Debian;
- Windows;
- Android в Termux.

Программа использует системный `ssh` как управляемый дочерний процесс. `tunnelctl` не запускает `ssh -f`, проверяет реальную работоспособность SOCKS через HTTP/HTTPS-запрос, перезапускает зависший процесс и поддерживает failover-группы.

## Основные возможности

- интерактивный выбор туннеля;
- прямой запуск профиля, алиаса или группы;
- первичный и повторный импорт команд `ssh -D` из историй shell, PowerShell, Far Manager и Total Commander;
- проверка SOCKS через реальный запрос;
- автоматическое переподключение и failover;
- ручное переключение активного VPS без поиска и убийства `ssh`-процессов;
- единое управление автозапуском через CLI;
- Linux user/system `systemd`, Планировщик заданий Windows и Termux:Boot.

## Быстрый старт

### Termux, Ubuntu, Debian

```bash
./start.sh
```

### Windows

```bat
start.cmd
```

Первый запуск проверяет окружение, создаёт конфигурацию, предлагает импортировать команды `ssh -D` и создать ярлыки.

## Повторный импорт

```bash
tunnelctl import
```

Команда повторно сканирует поддерживаемые истории и меню, показывает только ещё не настроенные туннели и сохраняет выбранные профили одним обновлением конфигурации.

## Конфигурация и логи

Конфигурация:

- Linux/Termux: `$XDG_CONFIG_HOME/tunnelctl/tunnelctl.json` или `~/.config/tunnelctl/tunnelctl.json`;
- Windows: `%APPDATA%\tunnelctl\tunnelctl.json`.

Логи и состояние:

- Linux/Termux: `$XDG_STATE_HOME/tunnelctl/` или `~/.local/state/tunnelctl/`;
- Windows: `%LOCALAPPDATA%\tunnelctl\`.

Управляющий канал:

- Linux/Termux: `$XDG_RUNTIME_DIR/tunnelctl/control.sock`, с fallback в каталог состояния;
- Windows: `\\.\pipe\tunnelctl-control`.

State-файл не содержит SSH-паролей, приватных ключей, токенов, имени пользователя или адреса VPS.

## Подключение

```bash
tunnelctl connect yandex --watch
```

Запускает профиль или алиас и поддерживает соединение.

```bash
tunnelctl connect auto --watch
```

Запускает failover-группу. Профили перебираются в порядке конфигурации.

```bash
tunnelctl connect yandex --no-watch
```

Выполняет один запуск без повторного подключения.

По умолчанию SOCKS слушает только `127.0.0.1:1080`. Прослушивание всех интерфейсов запрещено, пока оно явно не разрешено в конфигурации.

## Управление работающим туннелем

Команда `connect` в управляемом режиме остаётся владельцем дочернего `ssh`. Другие команды обращаются к нему через локальный управляющий канал текущего пользователя.

### Состояние

```bash
tunnelctl status
```

Показывает:

- запущен ли управляемый `tunnelctl`;
- PID;
- исходный профиль или группу;
- активный профиль;
- локальный SOCKS-адрес;
- время последней успешной проверки;
- путь к журналу;
- результат текущего health-check.

Открытый TCP-порт сам по себе не считается доказательством работоспособности SOCKS. Если управляющий процесс не найден, `status` отдельно сообщает, что принадлежность занятого порта `tunnelctl` не подтверждена.

### Ручное переключение

```bash
tunnelctl switch yandex
```

Переключает текущий туннель на профиль или алиас. Старый дочерний `ssh` завершается, порт освобождается, новый профиль запускается и проходит health-check.

```bash
tunnelctl switch next
```

Переходит к следующему профилю исходной failover-группы. Для одиночного профиля команда возвращает ошибку.

Если ручное переключение внутри группы не удалось, `tunnelctl` продолжает failover исходной группы. Для одиночного запуска выполняется возврат к предыдущему профилю.

### Остановка

```bash
tunnelctl stop
```

Корректно останавливает управляемый туннель и удаляет актуальный state-файл. Посторонние процессы, случайно занявшие SOCKS-порт, не завершаются.

## Автозапуск

Автозапуск управляется только через `tunnelctl`. Платформенные файлы и системные команды являются внутренней деталью реализации.

Устанавливаемый механизм всегда запускает:

```text
tunnelctl connect <профиль-или-группа> --watch --config <абсолютный-путь>
```

Внешний механизм отвечает только за запуск, остановку, состояние процесса и медленный restart-on-failure. Health-check, переподключение, backoff, ручное переключение и failover остаются внутри `tunnelctl`.

### Предварительный просмотр

```bash
tunnelctl autostart print auto
```

Показывает механизм, объект, режим, безопасную команду и полное содержимое создаваемого файла или задачи.

```bash
tunnelctl autostart install auto --dry-run
```

Показывает тот же план без создания каталогов, записи файлов, вызова `systemctl` или изменения Планировщика.

### Установка

```bash
tunnelctl autostart install auto
```

При необходимости можно указать конфигурацию явно:

```bash
tunnelctl autostart install auto --config /absolute/path/tunnelctl.json
```

До изменения системы команда проверяет, что профиль или группа существуют, вычисляет абсолютные пути и выводит план установки. Чужой существующий объект не перезаписывается.

### Состояние и управление

```bash
tunnelctl autostart status
tunnelctl autostart start
tunnelctl autostart stop
tunnelctl autostart remove
```

`autostart status` показывает состояние механизма запуска и отдельно пытается определить состояние управляемого туннеля. Состояние службы или задачи не считается доказательством работоспособности SOCKS; для полной проверки используется `tunnelctl status`.

Повторная установка с теми же параметрами идемпотентна. Повторное удаление отсутствующего объекта завершается понятным сообщением.

## Linux: systemd

### Пользовательская служба

По умолчанию создаётся:

```text
~/.config/systemd/user/tunnelctl.service
```

Используются команды:

```bash
systemctl --user daemon-reload
systemctl --user enable --now tunnelctl.service
systemctl --user status tunnelctl.service
systemctl --user start tunnelctl.service
systemctl --user stop tunnelctl.service
systemctl --user disable --now tunnelctl.service
```

Пользовательская служба обычно запускается после входа пользователя. Для запуска пользовательского manager сразу после загрузки без входа может потребоваться отдельно настроенный lingering. `tunnelctl` не выполняет `loginctl enable-linger` автоматически.

### Системная служба

```bash
tunnelctl autostart install auto --system --run-as tunneluser
```

Создаётся `/etc/systemd/system/tunnelctl.service`. Команду нужно запускать уже с правами root. `tunnelctl` не вызывает `sudo`, не запрашивает пароль и не повышает привилегии скрыто.

Параметр `--run-as` обязателен, чтобы SSH использовал домашний каталог, `known_hosts` и ключи ожидаемого пользователя.

Предварительный просмотр системной службы (`autostart print` или `--dry-run`) не требует root. Права администратора проверяются только перед изменением системы.

Для управления системной службой:

```bash
tunnelctl autostart status --system
tunnelctl autostart start --system
tunnelctl autostart stop --system
tunnelctl autostart remove --system
```

Изменяющие команды системного режима должны выполняться от root.

## Windows: Планировщик заданий

Создаётся задача:

```text
\tunnelctl\tunnelctl
```

Она запускается при входе текущего пользователя, использует абсолютный путь к `tunnelctl.exe`, не открывает интерактивное меню и включает restart-on-failure с задержкой. Команды `schtasks.exe` запускаются через `exec.Command` без оболочки.

Задача создаётся с уровнем `LeastPrivilege` для текущего пользователя. `tunnelctl` не вызывает UAC и не перезапускает `autostart install`, `remove`, справку или `--dry-run` от имени другой учётной записи.

Windows-служба на первом этапе не используется.

## Android / Termux: Termux:Boot

Создаётся управляемый сценарий:

```text
~/.termux/boot/tunnelctl
```

Сценарий использует абсолютные пути, `umask 077`, `termux-wake-lock` при наличии и медленный повтор только после ненулевого аварийного завершения `tunnelctl`.

Наличие каталога `~/.termux/boot` не доказывает, что приложение Termux:Boot установлено или настроено. `autostart status` пытается проверить пакет Android и выводит неопределённое состояние, если подтверждение получить нельзя.

Termux:Boot не является полноценным диспетчером служб. `autostart start` сообщает ограничение и рекомендует явный `tunnelctl connect <имя> --watch`. `autostart stop` использует управляющий канал и не выполняет ненадёжный поиск процессов по имени.

## Безопасность

- управляющий Unix socket создаётся с правами `0600` в каталоге `0700`;
- Windows named pipe запрещает удалённых клиентов и наследует локальный ACL процесса пользователя;
- state-файл создаётся с правами `0600`;
- автозапуск не содержит паролей, ключей, токенов или полного содержимого конфигурации;
- существующий объект без признака принадлежности `tunnelctl` не изменяется;
- посторонний процесс на SOCKS-порту не завершается автоматически;
- SOCKS по умолчанию слушает только loopback.
