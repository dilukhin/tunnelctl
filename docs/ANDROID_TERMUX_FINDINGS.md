# Findings: первый запуск tunnelctl в Termux

Дата проверки: 2026-07-23.

## Контекст

На Android в Termux был собран `tunnelctl 0.2.0`. Первичный мастер просканировал историю shell, импортировал найденные `ssh -D` команды и создал failover-группу `auto`. Пользователь ожидал три VPS, но полученная конфигурация не была работоспособной как целевой набор.

Проверка выполнялась на реальных SSH-серверах. Чувствительные данные и приватный ключ не копировались из каталога Termux.

## Подтверждённое исходное состояние

В конфигурации находилось шесть профилей из `.bash_history`:

- целевой сервер `v2` отсутствовал;
- профиль будущего `v0` не содержал `key`;
- сервер будущего `v3` присутствовал дважды, один раз без ключа и один раз со старым `~/.ssh/id_ed25519`;
- присутствовали посторонние серверы и пользователь `dilukhin`;
- группа `auto` автоматически включила все шесть импортированных профилей.

`doctor` корректно сообщил количество профилей, свободный SOCKS-порт и отсутствие одного из ключей. Однако он не проверял SSH-аутентификацию и фактическую работу SOCKS через каждый профиль.

## Причина

Мастер импортирует синтаксически подходящие команды `ssh -D` из истории, но история является журналом экспериментов, а не декларацией желаемого состояния. Текущая логика:

- считает разными профили с отличающимся ключом или другим полем;
- не знает, какие найденные серверы должны образовать целевую группу;
- создаёт `auto` из всех добавленных профилей;
- не проверяет профиль подключением перед сохранением или включением в группу.

Поэтому успешный bootstrap означает только создание валидного JSON, но не готовность туннелей.

## Применённая рабочая конфигурация

Конфигурация была сведена к трём профилям без поля `alias`:

| Профиль | Host | User | Port | Key |
|---|---|---|---:|---|
| `v0` | прямой IPv4 endpoint | `tunneluser` | 22 | `~/.ssh/tunnel_android` |
| `v2` | прямой IPv4 endpoint | `tunneluser` | 22 | `~/.ssh/tunnel_android` |
| `v3` | прямой IPv4 endpoint | `tunneluser` | 22 | `~/.ssh/tunnel_android` |

Группа `auto` содержит только `v0`, `v2`, `v3`. Имена являются именами профилей `tunnelctl`, а не SSH-алиасами; `sshproxy` формирует команду из прямых полей `host`, `user`, `port` и `key`.

## Проверка результата

Каждый профиль отдельно запускался командой `connect <profile> --watch`. Для всех трёх подтверждено:

1. запуск системного `ssh` с прямым IP;
2. открытие SOCKS5 на `127.0.0.1:1080`;
3. успешный HTTPS health-check через SOCKS;
4. корректное завершение тестового процесса и освобождение порта.

Итог:

```text
v0 OK
v2 OK
v3 OK
```

На отдельных соединениях OpenSSH выводил предупреждение об отсутствии post-quantum key exchange. Это серверное ограничение и не блокирует работу `tunnelctl`.

После успешного health-check встречались сообщения SSH `channel ... open failed`. Они не сорвали проверку и могли относиться к другим локальным подключениям к SOCKS. Если сообщения повторяются в штатной эксплуатации и совпадают с ошибками приложений, требуется отдельная диагностика адреса назначения.

## Продуктовые gaps

### P0. Команда проверки профиля

Нужна неинтерактивная команда, например:

```text
tunnelctl profile test <name>
tunnelctl doctor --connect
```

Она должна проверять:

- разрешение пути к ключу;
- SSH-аутентификацию с `BatchMode=yes` и `IdentitiesOnly=yes`;
- открытие SOCKS на временном свободном порту;
- HTTP/HTTPS health-check;
- освобождение порта и завершение дочернего `ssh`;
- безопасный результат без вывода пользователя и пути к ключу.

Критерий приёмки: профиль не считается рабочим только потому, что файл ключа существует.

### P0. Проверка импортированных профилей до включения в `auto`

Bootstrap должен отдельно показывать:

- импортирован;
- синтаксически валиден;
- ключ найден;
- подключение проверено;
- включён в failover-группу.

Непроверенные профили не следует автоматически добавлять в `auto` без отдельного подтверждения.

### P1. Улучшенная дедупликация истории

Для одинакового `user@host:port` с разными ключами мастер должен показывать конфликт, а не молча создавать несколько сгенерированных имён. Пользователь должен выбрать ключ и итоговое имя профиля.

### P1. Редактирование конфигурации через CLI

Нужны команды уровня:

```text
tunnelctl profile add
tunnelctl profile update
tunnelctl profile remove
tunnelctl group set-profiles
```

Это позволит исправлять импорт без ручной полной замены JSON.

### P1. Явный итог bootstrap

Финальное сообщение мастера должно различать:

- конфигурация создана;
- профили найдены;
- профили проверены;
- автозапуск установлен;
- управляемый туннель запущен.

Сейчас слово «Готово» может ошибочно восприниматься как подтверждение работоспособности серверов.

## Рекомендуемые тесты

- история содержит один endpoint с двумя разными ключами;
- история содержит устаревшие и недоступные серверы;
- часть профилей проходит test, часть отклоняется;
- `auto` создаётся только из явно выбранных профилей;
- `profile test` не оставляет `ssh`, state-файл или занятый SOCKS-порт;
- тест Termux с ключом, имя которого отличается от `id_ed25519`.

## Операционный вывод

До появления `profile test` после bootstrap необходимо запускать каждый профиль через `connect <name> --watch` и считать его рабочим только после успешного health-check. Конфигурация должна использовать прямые endpoint, если переносимость не должна зависеть от пользовательского `~/.ssh/config`.
